## 運用ルール

### 1. 各章の README.md 構成

各章の `README.md` は以下のセクションを持つ。**「重要な単語まとめ」は必ず章末に置く**。

1. `# Chapter N: タイトル`
2. `> Status: 執筆予定 / 執筆中 / 完了`
3. `## この章で学ぶこと`
4. `## 前提知識`
5. `## 本文`（教科書スタイルの読み物）
6. `## 手を動かすパート`（あれば）
7. `## つまずきポイント`
8. `## 次の章へ`
9. `## 重要な単語まとめ` ← **章の最後に必ず置く**

「重要な単語まとめ」には、その章で初めて出てきて、**かつ次章以降の理解に差し障る語** だけを一言定義つきで掲載する。網羅ではなく取捨選択。

### 2. 進捗が更新されたときの作業

章のステータスが変わったら（執筆予定 → 執筆中、執筆中 → 完了など）、**3 箇所を同期** する。

- このファイル（`CLAUDE.md`）の「現在の進捗」セクション
- ルート `README.md` の進捗チェックボックス
- 該当章の `README.md` 冒頭の `> Status: ...` 行

ユーザーから「進捗更新して」「`/init` で更新」と言われたら、上記 3 箇所を一括で見直す。

### 3. 進め方の方針

- **推測禁止**: 不明な点はユーザーに必ず質問する。`AskUserQuestion` を使う。「たぶんこうだろう」で進めない
- **過剰実装しない**: シンプルに保つ。フレームワーク・抽象化・将来のための拡張は不要
- **Python 中心**: Web 側で必要なときだけ JS/TS を使う
- **既存内容は壊さない**: ユーザーの追記指示があったときは、明示的に「変更してよい」と言われない限り既存セクションをそのまま残し、追加で書く
- **書籍の文章は引き写さない**: 自分の言葉で書き起こす。著作権・学習効果の両面で重要

### 4. 環境

- Python 3.11 以上を想定
- 仮想環境は `.venv/`
- 依存パッケージはルート `requirements.txt` に集約（章を進めるごとに追記）
- Web 側のコードを書くときは、章ディレクトリ内に必要に応じて `package.json` を置く

## これからの学習の流れ

### Chapter 2: ネットワーク（次に着手）

- IP アドレス、ポート番号、DNS、TCP/UDP、URL の構造
- 実装: Python で `nslookup` / `ping` / `curl` 相当を書く、`urllib.parse` で URL を分解、ソケットで直接通信
- ゴール: ブラウザでページが出るまでの「名前解決」「接続」を手で再現できる

### Chapter 3: HTTP ★実装重め

- メソッド・ステータスコード・ヘッダー・ボディ・Cookie・HTTPS
- 実装: 生ソケットで HTTP リクエストを手書き → `requests` → ブラウザ JS の `fetch` の順で同じことをやって比較
- ゴール: 「Web の通信はテキストの往復」だと体で分かる

### Chapter 4: データ形式

- HTML / JSON / CSV / MIME / 文字コード（UTF-8）
- 実装: JSON ↔ dict、CSV パース、拡張子から MIME 判定
- ゴール: API が返す JSON が「ただの文字列」だと分かる

### Chapter 5: Web アプリ ★実装重め

- フレームワークの役割、ルーティング、REST API、CORS
- 実装: Flask または FastAPI で簡易 API → ブラウザ JS の `fetch` から叩く → CORS エラーを意図的に出して直す
- ゴール: 自分で API を立てて、自分で叩ける

### Chapter 6: セキュリティ

- SQL インジェクション、XSS、CSRF、認証と認可、HTTPS の意義
- 実装（ローカル限定）: 脆弱なコードを書いて攻撃が通ることを確認 → 修正して通らなくする
- ゴール: 「なぜそうすべきか」が攻撃側の視点も含めて分かる

### Chapter 7: 構築・運用

- Web サーバー / アプリケーションサーバー / DB / 負荷分散 / クラウドの位置づけ
- 実装: Chapter 5 で作った API を実際にクラウドへデプロイして公開する
- ゴール: 「自分のコードが世界に公開される」までのフルパスを一度通す

## よくある作業の Tips

- **章の本文を書く前に**: その章で扱うトピックを箇条書きで列挙 → ユーザーに「この範囲で良いか」を確認 → 書き始める
- **コードを書く前に**: 章本文に書いた説明と、コードが目指す挙動が一致しているか確認
- **章を完了にするとき**: 「重要な単語まとめ」が章末にあるか、ルート README とこの CLAUDE.md の進捗が同期しているかを確認
