# CLAUDE.md — web-basics-handbook

このファイルは、Claude Code がこのプロジェクトで作業するときの **運用ルール** と **進捗・学習計画** をまとめたものです。新しいセッションを始めるとき、Claude Code はまずこの文書を読んでから作業を開始してください。

進捗や方針が更新されたら、このファイルも `/init` で更新します（手動編集でも可）。

## 運用ルール

### 1. 各章の README.md 構成

各章の `README.md` は以下のセクションを持つ。**「重要な単語まとめ」は必ず章末に置く**。

1. `# Chapter N: タイトル`
2. `> Status: 執筆予定 / 執筆中 / 完了`
3. `## この章で学ぶこと`
4. `## 前提知識`
5. `## 本文`（教科書スタイルの読み物）
6. `## 手を動かすパート`（あれば）
7. `## つまずきポイント`
8. `## 次の章へ`
9. `## 重要な単語まとめ` ← **章の最後に必ず置く**

「重要な単語まとめ」には、その章で初めて出てきて、**かつ次章以降の理解に差し障る語** だけを一言定義つきで掲載する。網羅ではなく取捨選択。

### 2. 進捗が更新されたときの作業

章のステータスが変わったら（執筆予定 → 執筆中、執筆中 → 完了など）、**3 箇所を同期** する。

- このファイル（`CLAUDE.md`）の「現在の進捗」セクション
- ルート `README.md` の進捗チェックボックス
- 該当章の `README.md` 冒頭の `> Status: ...` 行

ユーザーから「進捗更新して」「`/init` で更新」と言われたら、上記 3 箇所を一括で見直す。

### 3. 進め方の方針

- **推測禁止**: 不明な点はユーザーに必ず質問する。`AskUserQuestion` を使う。「たぶんこうだろう」で進めない
- **過剰実装しない**: シンプルに保つ。フレームワーク・抽象化・将来のための拡張は不要
- **Python 中心**: Web 側で必要なときだけ JS/TS を使う
- **既存内容は壊さない**: ユーザーの追記指示があったときは、明示的に「変更してよい」と言われない限り既存セクションをそのまま残し、追加で書く
- **書籍の文章は引き写さない**: 自分の言葉で書き起こす。著作権・学習効果の両面で重要
- **新規の専門用語には必ず身近な実例を添える**: 章で初めて登場する用語は、ユーザーが日常的に使っている / 見ているもの（YouTube・LINE・Discord・大学 Wi-Fi・スマホ設定画面・`http://localhost:3000` のような開発用 URL・Wikipedia の見出しリンクなど）を引き合いに出して説明する。理解促進のための必須ルール。`## 重要な単語まとめ` の表にも「身近な実例」列を含める

### 4. 環境

- Python 3.11 以上を想定
- 仮想環境は `.venv/`
- 依存パッケージはルート `requirements.txt` に集約（章を進めるごとに追記）
- Web 側のコードを書くときは、章ディレクトリ内に必要に応じて `package.json` を置く

## 質問ログと番外編（補強ページ）

学習中、ユーザーが Claude Code に「わからない」を聞いてきたら、**回答するだけでなく必ず記録** する。記録が一定数たまった時点で補強ページ作成を提案する。

### 振る舞いルール

1. **質問の検出**: 下記の検出基準に当てはまる発言が来たら処理を発動
2. **回答 + 同時記録**: ユーザーへの回答を出すのと **同じターン** で `extras/qa-log.md` に 1 エントリ追記する（沈黙で記録だけ進めない、回答だけして記録忘れもなし）
3. **関連章の推定**: 質問内容から関連する章を推定（Ch1〜Ch7、または「全体」）
4. **3 件たまったら提案**: 同じ関連章で「補強済み: ☐」のエントリが **3 件以上** たまったら、回答の末尾で「○○について 3 件たまりました。補強ページ作りますか？」と **一度だけ** 提案する。断られたら提案カウンタを記録のうえ、しばらく出さない（5 件たまったら再提案）
5. **承認されたら**: `extras/supplements/NNN-タイトル.md` を新規作成し、qa-log の該当エントリを統合・整理。元の qa-log エントリは「補強済み: ☑」に更新し、`extras/supplements/README.md` の一覧表にも追加する
   - **補強ページの単位**: 章（Ch）ごとに 1 ファイルにまとめる。例: Ch1 の質問が複数あれば `001-ch1-補強.md` 1 ファイルに集約する

### 質問の検出基準

**記録する**:
- 「○○がわからない」「○○とは？」「○○について教えて」「○○ってどういう意味？」「○○と○○の違いは？」など、**学習目的の質問**
- 章本文を読んだあとの「ここの意味が分からなかった」系の聞き直し

**記録しない**:
- 「Ch3 書いて」「ファイル作って」のような作業指示
- ハンドブックの執筆方針・運用についての相談
- 単なる雑談や状況確認

迷ったら記録する側に倒す。あとで整理できる。

### 記録フォーマット（再掲）

```
### YYYY-MM-DD: <質問の見出し>
- 関連章: ChN
- 質問: <要旨 1 行>
- 回答要点:
  - <要点 1>
  - <要点 2>
  - <要点 3>
- 補強済み: ☐
```

### 補強ページ作成時のルール

- ファイル名: `extras/supplements/NNN-タイトル.md`（NNN は連番、001 から）
- 本編の章と同じく、新規の専門用語には身近な実例を添える
- 該当の qa-log エントリの「補強済み」を `☑` にする
- `extras/supplements/README.md` の一覧表に行を追加する
- 関連章の本文（`chapters/0N-XXX/README.md`）は書き換えない

## よくある作業の Tips

- **章の本文を書く前に**: その章で扱うトピックを箇条書きで列挙 → ユーザーに「この範囲で良いか」を確認 → 書き始める
- **コードを書く前に**: 章本文に書いた説明と、コードが目指す挙動が一致しているか確認
- **章を完了にするとき**: 「重要な単語まとめ」が章末にあるか、ルート README とこの CLAUDE.md の進捗が同期しているかを確認
- **質問に答えるとき**: 学習目的の質問なら、回答と同じターンで `extras/qa-log.md` に追記することを忘れない
