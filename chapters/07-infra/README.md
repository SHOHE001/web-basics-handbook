## 本文

### 1. ローカルと本番のあいだに何があるか

Chapter 5 で `uvicorn app:app --reload` を叩いて、`http://127.0.0.1:8000` で API を立てました。これを「友達のスマホから `https://my-todo.example.com` で叩ける」状態にするには、次の壁を越える必要があります。

1. **公開された IP（とドメイン）が要る** — `127.0.0.1` は自分の PC からしか見えない
2. **常時起動が要る** — PC を閉じたら止まる
3. **HTTPS が要る** — 今どきブラウザは HTTP のサイトに警告を出す
4. **複数の同時アクセスに耐える必要** — 1 ユーザーなら uvicorn 単体でも回るが、100 ユーザーは別の話
5. **デプロイの手順が要る** — コードを更新したら本番にも反映したい
6. **障害に気付く仕組みが要る** — 寝てる間にサーバーが落ちて朝気付く、では遅い

この章は、それを支える **登場人物（サーバーの種類）** と **流れ（デプロイ・運用）** の地図です。

### 2. Web サーバーとアプリケーションサーバー

「Web サーバー」と「アプリケーションサーバー」は紛らわしいですが、別の役割を持つ別のソフトです。

| | <mark>**Web サーバー**</mark> | <mark>**アプリケーションサーバー**</mark> |
|---|---|---|
| 役割 | リクエストを受け取って、静的ファイルを返す or アプリに渡す | Python / Ruby / Java などの **アプリのコード** を実行する |
| 代表例 | **Nginx**、Apache | **uvicorn**、**gunicorn**、Puma、Tomcat |
| 仕事の例 | HTML / 画像 / CSS / JS の配信、HTTPS の終端、ロードバランス | `@app.get("/items")` のような Python の関数を呼ぶ |

実運用ではこれらを **2 段構え** にすることが多い:

```
ブラウザ
  │  HTTPS
  ▼
[Nginx]                ← Web サーバー（HTTPS を解く、静的ファイル直接返す、それ以外は↓へ）
  │  HTTP
  ▼
[uvicorn + FastAPI]    ← アプリケーションサーバー
  │
  ▼
[PostgreSQL]           ← DB サーバー
```

**なぜ分けるか**:
- 静的ファイル（CSS、画像）は **Nginx に直接返させる方が速い**。アプリを通すと無駄
- HTTPS の証明書まわりを **Nginx に任せて**、アプリは平文 HTTP だけ気にすればよくなる
- 同じ Nginx の裏に **複数のアプリインスタンス** をぶら下げて、負荷分散できる

**実例**: あなたが日常で見ている大規模サイト（YouTube、Amazon、Twitter）は、ほぼ全て複数台のサーバーに Nginx 系のフロントが立っていて、その裏で何十・何百ものアプリプロセスが動いている。

### 3. データベースサーバー

DB は **別プロセス、できれば別マシン** で動かすのが定石。

- **DB はディスクとメモリを大量に食う**: アプリと同じマシンに乗せると競合する
- **DB は再起動コストが高い**: アプリをデプロイで再起動するたびに DB も巻き添えにしたくない
- **複数のアプリインスタンスから同じ DB を共有する** ことが多い: 別プロセスにしておけば素直

代表的な DB:

| 種類 | 例 | 用途 |
|---|---|---|
| <mark>**RDB（リレーショナル DB）**</mark> | PostgreSQL, MySQL, SQLite | テーブルと SQL。ほとんどの Web アプリのデフォルト |
| <mark>**NoSQL（ドキュメント DB）**</mark> | MongoDB, DynamoDB | JSON 風の柔らかいデータ。スキーマがゆるい |
| <mark>**KVS（キーバリュー）**</mark> | Redis, Memcached | キャッシュ・セッション・キュー |
| 検索エンジン | Elasticsearch, OpenSearch | 全文検索 |

迷ったら **PostgreSQL**。困ったら **Redis を横に置く**。これでだいたいの Web アプリは回ります。

### 4. 負荷分散と冗長化

#### 4-1. ロードバランサ

ユーザーが増えたとき、1 台のアプリサーバーを **「もっと強いマシン」に置き換える**（スケールアップ）には限界があります。普通は **「同じアプリを何台も並べる」**（スケールアウト）方向に伸ばします。

そのとき、ユーザーからのリクエストを **どの台に振るか決める** のが <mark>**ロードバランサ（LB）**</mark>。

```
       ┌── [app #1]
[LB] ──┼── [app #2]   ← どこへ振るかは LB が決める
       └── [app #3]
```

振り分け方:
- **ラウンドロビン**: 順番に振る
- **最少接続**: 今いちばん暇な台に振る
- **IP ハッシュ**: 同じ IP のユーザーは同じ台に振る（セッションの一貫性のため）

#### 4-2. 冗長化

「1 台あれば動く」は危ない。<mark>**1 台が壊れても他で受けられる**</mark>ように、最初から複数台で組むのが <mark>**冗長化**</mark>。LB の裏に app を 2 台以上置く、DB をプライマリ + レプリカで組む、など。

「動くものを 1 つ作る」と「落ちないように 2 つ以上で作る」は **設計のレベルが 1 段違う**、というのが運用の現実です。

**実例**: 大規模サイトが「メンテナンスのお知らせ」を出さずに常時動いていられるのは、こうした冗長化のおかげ。1 台ずつ順に止めてアップデートしても、全体としてはサービスが止まらない。

### 5. クラウドの地図

「自分でサーバーを買って、OS を入れて、Nginx と Python を入れて、…」は今でも可能ですが、ほとんどの場合は **クラウドサービス** を使います。クラウドのレイヤをざっくり分けると:

| レイヤ | 自分が見るもの | 例 |
|---|---|---|
| <mark>**IaaS**（Infrastructure as a Service）</mark> | 仮想マシン（OS から自分で面倒みる） | AWS EC2, GCP Compute Engine |
| <mark>**PaaS**（Platform as a Service）</mark> | アプリのコードだけ。サーバーは見えない | Heroku, Render, Fly.io, Railway |
| <mark>**サーバーレス / FaaS**</mark> | 1 つの関数だけ。リクエスト時だけ起きる | AWS Lambda, Cloud Functions |
| **フロントエンド特化** | ビルド済みフロントを配信 + サーバーレス関数 | Vercel, Netlify, Cloudflare Pages |
| **マネージド DB** | DB だけ預ける | AWS RDS, Supabase, PlanetScale |

学習者・個人開発者向けには、**Render** / **Fly.io** / **Vercel** などの PaaS 系が圧倒的に楽。「Git に push したら勝手にデプロイ・HTTPS 自動・無料枠あり」というセットになっています。

> [!TIP]
> いきなり「AWS で 1 から組む」は学習コストが高すぎてだいたい挫折します。<mark>**最初は PaaS で「自分のコードが世界から見える」体験を 1 回通す**</mark>のがおすすめ。AWS の細かい部品（VPC、IAM、ALB、…）は、必要になってから学んでも遅くない。

### 6. デプロイの流れ

「ローカルで動いた → 世界から見える」までの典型的な流れ。

1. **コードを Git に置く**（GitHub など）
2. **デプロイ先サービスにリポジトリを連携**（Render なら「Connect GitHub repository」を選ぶ）
3. **起動コマンドを指定**（FastAPI なら `uvicorn app:app --host 0.0.0.0 --port $PORT`）
4. **環境変数を設定**（DB のパスワード、API キーなど）
5. **デプロイ実行**: サービスがビルド・起動して URL を払い出す
6. **`https://<your-app>.onrender.com` を開いて動作確認**

これに加えて運用フェーズでは:

- **CI/CD**: <mark>**`main` ブランチに push したら自動でテスト → 自動でデプロイ**</mark>を流す（GitHub Actions など）
- **環境分離**: `development` / `staging` / `production` の 3 つを用意するのが定番
- **ロールバック**: 新バージョンで問題が出たら 1 つ前に戻せるようにしておく

#### `host` の話

ローカルでは `uvicorn app:app` だけで起動できますが、本番では `--host 0.0.0.0` を必ず付けます。

- `127.0.0.1` だけにバインド = 自分の PC からしか見えない（Chapter 2 でやった）
- `0.0.0.0` にバインド = <mark>**マシンが持つ全 IP で待ち受ける**</mark> = 外部からアクセス可能

「`127.0.0.1` で起動したのにクラウド上でアクセスできない」は超頻出のハマりどころ。

### 7. ログと監視

本番運用は **「動いていない時間に気付けるかどうか」** で半分決まります。

#### ログ

- アプリは何が起きたかを **標準出力 / 標準エラーに print** する。クラウドのサービスがそれを集めて見せてくれる
- print 文ではなく <mark>**`logging` モジュール** を使い、レベル（DEBUG / INFO / WARNING / ERROR）を意識</mark> する
- **構造化ログ**（JSON で書く）にしておくと、後で検索・集計しやすい

#### モニタリング

- **死活監視**: 一定間隔で `GET /healthz` を叩いて、200 が返るかチェック。落ちたら通知（Slack / メール）
- **メトリクス**: CPU 使用率、メモリ、レスポンスタイム、エラー率を時系列で記録 → グラフで眺める
- **アラート**: 「エラー率が 5% を超えたら通知」など、しきい値ベースで自動通知

代表的なツール: **Sentry**（エラー追跡）、**Datadog / Grafana / Prometheus**（メトリクス）、**Cloudflare Healthchecks**。

PaaS の多くは標準でログとメトリクスを見せてくれるので、最初はそれで十分。

### 8. 全体の地図（最後に）

ここまでの登場人物をひとつの図にまとめると:

```
  ユーザー
    │  HTTPS
    ▼
  [CDN] (オプション。Cloudflare など)
    │
    ▼
  [ロードバランサ]
    │     │     │
  [app1][app2][app3]    ← FastAPI on uvicorn × 複数台
    │     │     │
    ▼     ▼     ▼
       [DB プライマリ] ── 同期 ── [DB レプリカ]
       [Redis（キャッシュ・セッション）]

  上の全部を:
    - Git にコードを置き
    - CI でテストし
    - CD で本番にデプロイし
    - ログ・メトリクス・アラートで監視する
```

**Chapter 1 で見た「ブラウザに URL を入れたあと何が起きるか」が、ここまで来てやっと全部つながります。** Chapter 2 の DNS、Chapter 3 の HTTP、Chapter 4 の JSON、Chapter 5 の Web アプリ、Chapter 6 の HTTPS と防御。それぞれの章で見た部品が、本番ではこの地図上のどこかに乗っている、と分かれば全章完走です。

## つまずきポイント

- <mark>**「ローカルで動いた」と「本番で動く」は別問題**</mark>: 環境変数、`host=0.0.0.0`、ファイルパス、タイムゾーン、文字コード … 違いは多い
- <mark>**`127.0.0.1` ではなく `0.0.0.0` で起動する**</mark>: 本番デプロイで最初にハマる
- <mark>**環境変数で秘密情報を渡す**</mark>: パスワードや API キーをコードに直書きしない。Git にも入れない（`.env` は `.gitignore`）
- **ログは「未来の自分が読むメモ」**: 何かあったときログだけが手がかり。意味のある情報を出す
- **依存ライブラリは固定する**: `requirements.txt` でバージョンを固定しないと、本番ビルド時に勝手にメジャーアップデートされて壊れることがある
- **DB のマイグレーション**: スキーマ変更は手順を間違えると本番データを壊す。Alembic などのツールで管理する
- **コストの監視**: クラウドは「気付いたら数万円」が普通に起きる。請求アラートを必ず設定する
- **バックアップ**: DB のバックアップは取る。「取ってるつもり」を「復元できる」までテストする

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **Web サーバー** | HTTP を受けて静的ファイル配信 / アプリへ転送するソフト | Nginx, Apache |
| **アプリケーションサーバー** | アプリのコード（Python など）を実行するサーバー | uvicorn, gunicorn |
| **リバースプロキシ** | クライアントの代わりに裏のサーバーへ取り次ぐ | Nginx を FastAPI の前に置く構成 |
| **RDB / NoSQL / KVS** | DB の主要な種類（テーブル / ドキュメント / キーバリュー） | PostgreSQL, MongoDB, Redis |
| **ロードバランサ（LB）** | リクエストを複数台に振り分ける装置 | AWS ALB, Cloudflare Load Balancing |
| **スケールアップ / スケールアウト** | 1 台を強くする / 台数を増やす | サーバーのプラン変更 / 台数追加 |
| **冗長化** | 1 台壊れても全体は動く構成にする | DB のプライマリ + レプリカ |
| **IaaS / PaaS / FaaS** | クラウドの抽象化レベル | AWS EC2 / Render / AWS Lambda |
| **CDN** | 静的コンテンツを世界中のエッジから配信する仕組み | Cloudflare, CloudFront |
| **環境変数** | コードに書かずに外から渡す設定値 | DB パスワード、API キー |
| **CI/CD** | テスト・ビルド・デプロイの自動化パイプライン | GitHub Actions, GitLab CI |
| **`0.0.0.0`** | マシンの全 IP で待ち受けるバインドアドレス | クラウドで API を起動するときの定番 |
| **死活監視 / ヘルスチェック** | サービスが生きているか定期的に確認する | `GET /healthz` を 1 分ごとに叩く |
| **`logging`（構造化ログ）** | レベル付きで、機械でも読めるログ | JSON 形式で標準出力に流す |
| **Sentry / Datadog** | エラー追跡・メトリクス監視の代表ツール | 本番のエラーを Slack に通知 |
