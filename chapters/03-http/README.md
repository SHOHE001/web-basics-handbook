## 本文

### 1. HTTP は「テキストの往復」

衝撃的かもしれませんが、HTTP の通信は <mark>**ただのテキストの行のやり取り**</mark> です。バイナリでもなく、暗号でもなく（暗号化は TLS の仕事で、それは HTTP の外側の話）、人間が読めば普通に意味が分かる英語の行が往復しているだけです。

クライアントからサーバーへ送られるテキストの塊を **HTTP リクエスト**、サーバーからクライアントへ返ってくる塊を **HTTP レスポンス** と呼びます。Chapter 2 で確立した TCP コネクションの上に、このテキストの塊を流すだけ。それが HTTP です。

**実例**: ブラウザで適当なページを開いて `F12` で開発者ツールを出し、`Network` タブで通信を 1 つ選んで `Headers` を見ると、`GET /xxx HTTP/1.1` のような行とその下にズラッと並ぶヘッダーが見えます。あれが今この瞬間にあなたのブラウザがサーバーに送ったテキストそのものです。

### 2. リクエストの構造

HTTP リクエストは、決まった順番でテキストが並んでいます。

```
GET /search?q=web HTTP/1.1            ← リクエストライン
Host: www.example.com                 ┐
User-Agent: Mozilla/5.0 ...           │ ヘッダー
Accept: text/html                     │（何行でもいい、改行で区切る）
Cookie: sessionid=abc123              ┘
                                      ← 空行（必須、ここで「ヘッダー終わり」と伝える）
（ボディ。GET なら空、POST ならフォーム内容や JSON などが入る）
```

3 つのパートに分かれています。

1. **リクエストライン**: `メソッド` + `パス` + `HTTP バージョン`
2. **ヘッダー**: `名前: 値` の行を必要なだけ並べる
3. **ボディ**: サーバーに送るデータ本体。GET では普通空っぽ、POST / PUT で使う

ポイントは **ヘッダーとボディの間に必ず空行を 1 つ入れる** こと。これがないとサーバーはヘッダーがどこで終わるか分からず、リクエストが壊れます。

> [!CAUTION]
> 生ソケットで HTTP を書くとき、**改行は LF (`\n`) ではなく CRLF (`\r\n`)** で送る。そして **ヘッダーの最後はもう 1 つ空行（`\r\n\r\n`）で締める**。この 2 点を忘れると `01_raw_http.py` を書き換えたとき必ず詰まる。

**実例**: ブラウザでログインフォームに ID/パスワードを入れて「ログイン」を押すと、`POST /login HTTP/1.1` から始まるリクエストが飛んで、ボディに `username=alice&password=...` のような文字列が入っています。`F12 → Network → Payload` で実際に見られます。

### 3. レスポンスの構造

リクエストとほぼ対称です。違いは 1 行目だけ。

```
HTTP/1.1 200 OK                       ← ステータスライン
Content-Type: text/html; charset=utf-8 ┐
Content-Length: 1234                   │ ヘッダー
Set-Cookie: sessionid=xyz789           ┘
                                       ← 空行
<!DOCTYPE html>                        ┐
<html>...                              │ ボディ（HTML や JSON や画像など）
</html>                                ┘
```

1. **ステータスライン**: `HTTP バージョン` + `ステータスコード` + `理由フレーズ`
2. **ヘッダー**: リクエストと同じ形式
3. **ボディ**: 実際の中身（HTML、JSON、画像のバイナリ、何でも）

**実例**: 開発者ツールで通信を選んで `Response` タブを開くと、そこに表示されているテキストがまさにレスポンスのボディです。JSON が返ってくる API なら整形された JSON が、HTML なら HTML タグの塊が見えます。

### 4. メソッド

リクエストラインの先頭にくる **メソッド** は、「サーバーに何をしてほしいか」を表します。

| メソッド | 役割 | 例え |
|---|---|---|
| **GET** | 取得 | ページや画像を「読む」 |
| **POST** | 作成・送信 | フォーム送信、新規投稿 |
| **PUT** | 置き換え | リソース全体を新しい内容で上書き |
| **PATCH** | 部分更新 | 一部のフィールドだけ更新 |
| **DELETE** | 削除 | リソースを消す |
| **HEAD** | ヘッダーだけ取得 | サイズや更新日時だけ知りたい |
| **OPTIONS** | サーバーが対応しているメソッドを聞く | CORS のプリフライトでよく使う |

**実例**:
- **GET**: あなたが Twitter / X のタイムラインを開く、Wikipedia の記事を読む、YouTube のサムネイル画像を読み込む。全部 GET
- **POST**: ツイートを投稿する、Amazon で「カートに入れる」、Google フォームに送信する
- **PUT / PATCH**: プロフィールを編集する。「全部書き換える」が PUT、「名前だけ更新する」が PATCH
- **DELETE**: ツイートを消す、メールをアーカイブする

#### GET と POST の意味の違い

ここを誤解する人が多いので強調しておきます。

- <mark>**GET は「副作用なし（safe）」**: サーバーの状態を変えてはいけない。読むだけ</mark>
- <mark>**POST は副作用あり**: 実行するたびにサーバーの状態が変わる可能性がある（投稿が増える、注文が確定する）</mark>

ここでの「安全（safe）」は HTTP 仕様上の専門用語で、日常語の「セキュリティ的に安全」とは別物です。むしろログイン情報を GET で送ると URL に残るので **セキュリティ的にはかえって危ない**。本質は <mark>**副作用があるかないか**</mark>。検索フォームは GET（`?q=...`）でいい、ログインフォームや投稿フォームは POST。

**実例**: ブラウザの「戻る」ボタンを押したとき、ページが POST で取得されたものだと「フォームを再送信しますか？」と聞かれることがあります。あれは POST が「もう一度実行すると別の結果になるかもしれない」とブラウザが知っているからです。

#### 冪等性（べきとうせい）

`PUT` `DELETE` `GET` は **何度実行しても、サーバーの最終的な状態が同じ**（DELETE を 2 回実行しても「消えている」状態は変わらない。2 回目のレスポンスが `404` になっても、サーバーの状態としては同じ）。これを <mark>**冪等（idempotent）**</mark> と言います。`POST` は冪等ではない（投稿が 2 件できる）。設計のときによく出てくる概念です。

### 5. ステータスコード

レスポンスの 1 行目の **3 桁の数字**。サーバーが「結果はこうでした」と返す簡潔な合図です。

| 範囲 | 意味 | 代表例 |
|---|---|---|
| **1xx** | 情報 | ほぼ使わない |
| **2xx** | 成功 | `200 OK`, `201 Created`, `204 No Content` |
| **3xx** | リダイレクト | `301 Moved Permanently`, `302 Found`, `304 Not Modified` |
| **4xx** | クライアント側のエラー | `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `429 Too Many Requests` |
| **5xx** | サーバー側のエラー | `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable`, `504 Gateway Timeout` |

「クライアント側のエラー」は **サーバーから見た分類** で、必ずしも利用者の操作ミスとは限らない（404 はリンク切れでも出るし、429 はサーバー側のしきい値次第）。リンク切れを踏んで 404 が出たからといって「自分が悪い」とは限らない、という温度感で読みます。

#### 覚えておくと便利な区別

- **404 Not Found**: 「そんなページはない」。あなたも一度は見たことがある定番のエラー画面
- **403 Forbidden**: 「そのページはあるけど、あなたには見せない」。権限不足
- **401 Unauthorized**: 「そもそも誰だか名乗ってない」。ログインが必要
- **500 vs 502 vs 503**:
  - `500` はアプリケーションのバグ（例外がそのまま漏れた、など）
  - `502` は前段のサーバー（リバースプロキシ・ロードバランサ）がアプリに繋げなかった
  - `503` はサーバーが「今は受け付けない」と意図的に拒否（メンテ・過負荷）

**実例**:
- **404**: 古いブログ記事のリンクをクリックすると出てくる「ページが見つかりません」
- **429**: API を短時間で叩きすぎると「Too Many Requests」と返される（Twitter/X や OpenAI の API でおなじみ）
- **301 vs 302**: `http://example.com` を打ったら自動で `https://...` に飛ぶ、あれはリダイレクトです。「永久に変わったよ」が `301`、「とりあえず今は別の場所」が `302`

#### 「ステータスコードは嘘をつけてしまう」

> [!CAUTION]
> サーバー側の実装次第で、本当はエラーなのに `200 OK` を返したり、その逆もできてしまう。<mark>**「`200` が返ってきた＝成功」と決めつけない**</mark>。レスポンスのボディも合わせて確認するのが堅実。とくに古い API では `200 OK` のボディの中に `{"error": "..."}` が入っているパターンがある。Ch5 で自分の API を書くときに必ず思い出すこと。

### 6. ヘッダー

リクエストにもレスポンスにも付く **メタ情報**。「ボディの中身が何か」「ブラウザの種類」「キャッシュしていいか」など、本文の脇に添える注記です。

#### よく出てくるヘッダー

| ヘッダー | 方向 | 役割 |
|---|---|---|
| **Host** | 要求 | どのドメイン宛てか（バーチャルホスト用、HTTP/1.1 で必須） |
| **User-Agent** | 要求 | ブラウザの種類・OS の情報 |
| **Accept** | 要求 | 「こういう形式で返してほしい」（`text/html`, `application/json` など） |
| **Accept-Language** | 要求 | 「日本語があるならそれをくれ」 |
| **Cookie** | 要求 | 前回サーバーが渡した Cookie をそのまま送り返す |
| **Authorization** | 要求 | 認証情報（API トークンなど） |
| **Content-Type** | 両方向 | ボディの形式（`application/json` など） |
| **Content-Length** | 両方向 | ボディのバイト数 |
| **Set-Cookie** | 応答 | Cookie をクライアントに保存させる指示 |
| **Location** | 応答 | リダイレクト先 URL（3xx と一緒に使う） |
| **Cache-Control** | 応答 | キャッシュの方針 |

**実例**:
- スマホで Google にアクセスするとモバイル版が出るのは、`User-Agent` を見てサーバーが「これはスマホだな」と判断しているから
- 日本からアクセスして日本語のページが返るのは、`Accept-Language: ja` を送っているから
- YouTube の動画ページを開くと自動でログイン状態になっているのは、ブラウザが保存していた `Cookie` を送ったから

#### `Content-Type` は超重要

<mark>**ボディが何の形式かを宣言するヘッダー**</mark>。これが間違っているとブラウザはボディをうまく解釈できません。

- `text/html; charset=utf-8` ← HTML
- `application/json` ← JSON
- `text/css` ← CSS
- `image/png` ← PNG 画像
- `application/x-www-form-urlencoded` ← フォーム送信の標準形式
- `multipart/form-data` ← ファイルアップロード

**実例**: API を作るときに `Content-Type: application/json` を付け忘れると、`axios` のような `Content-Type` を見て自動でパースを切り替えるライブラリが「これは JSON じゃない」と判断して dict に変換せず生文字列のまま渡してくる、というのは新人エンジニアの定番ミスです（ブラウザの素の `fetch().then(r => r.json())` は逆に Content-Type を見ずに強制パースするので、こちらは別の意味で混乱の元）。

### 7. Cookie とセッション

> [!CAUTION]
> HTTP には <mark>**状態を覚える機能がない**</mark>（=「ステートレス」）。リクエスト 1 回ごとに完全に独立。サーバーは「さっき来たのと同じ人かどうか」を本来は知らない。**この前提を忘れると Cookie・セッション・認証の話が全部分からなくなる**。

でも、Web サイトはあなたを覚えていますよね。Amazon はカートの中身を保持しているし、Twitter / X は次に開いてもログイン状態のまま。これは **Cookie** という仕組みで状態を擬似的に保持しているからです。

#### Cookie の流れ

```
1 回目:
クライアント → サーバー  : GET /login + ID/PW
サーバー → クライアント  : 200 OK + Set-Cookie: sessionid=abc123

2 回目以降:
クライアント → サーバー  : GET /mypage + Cookie: sessionid=abc123
サーバー  : 「sessionid=abc123 はアリスだな」と思い出す
```

- サーバーが `Set-Cookie` ヘッダーで「これを覚えておいて」と渡す
- ブラウザはそれをローカルに保存する
- 次回からは同じドメインへのリクエストに `Cookie` ヘッダーで自動で添付される

#### Cookie の属性

`Set-Cookie` には属性を付けてセキュリティや挙動を制御します。

| 属性 | 意味 |
|---|---|
| **Expires / Max-Age** | 有効期限。なければブラウザを閉じるまで |
| **Domain** | どのドメイン宛てに送るか |
| **Path** | どのパス以下で送るか |
| **Secure** | HTTPS のときだけ送る |
| **HttpOnly** | JavaScript からは読めない（XSS 対策） |
| **SameSite** | クロスサイトリクエストで送るか（CSRF 対策） |

**実例**: 開発者ツールの `Application → Cookies` を開くと、今そのサイトに保存されている Cookie の一覧と、各属性が全部見えます。ログイン中の Twitter / X や Google を見ると、`HttpOnly` と `Secure` が付いたセッション ID が必ず入っています。

#### セッションとの関係

「セッション」とは、サーバー側で **同じユーザーからの連続したアクセス** をひとまとめにする概念です。Cookie はそのセッションを識別する「整理券」のような役割。

サーバーは `sessionid=abc123` を見て、自分の DB やメモリから「abc123 はアリスのカート」「アリスは管理者権限」みたいな情報を引っ張ってきます。

### 8. HTTP のバージョン

| バージョン | 特徴 |
|---|---|
| **HTTP/1.0** | 1996 年。リクエストごとに TCP 接続を張り直す |
| **HTTP/1.1** | 1997 年。接続を使い回せる（Keep-Alive）。今でも現役 |
| **HTTP/2** | 2015 年。1 本の接続で並列にやり取り（多重化）、ヘッダー圧縮 |
| **HTTP/3** | 2022 年。TCP ではなく QUIC（UDP の上）。接続確立が速く、パケットロスに強い |

HTTP/2 以降は **バイナリプロトコル** になっていてテキストで読めません。が、概念（メソッド・ステータス・ヘッダー）は HTTP/1.1 と同じです。だから「HTTP の勉強は HTTP/1.1 で十分」と言われます。

**実例**: 開発者ツールの `Network → Protocol` 列を出すと、各リクエストが `h2`（HTTP/2）か `h3`（HTTP/3）か `http/1.1` で動いているか分かります。Google・YouTube・Cloudflare 配下のサイトは `h3` が多い。

### 9. HTTPS と TLS

<mark>**HTTPS は HTTP + TLS**</mark> です。HTTP 自体は変わらず、TCP との間に **TLS という暗号化レイヤ** をはさんでいるだけ。

```
HTTP   :  HTTP の文字列  → TCP                → IP
HTTPS  :  HTTP の文字列  → TLS で暗号化  → TCP → IP
```

#### TLS が解決すること

1. **盗聴防止**: 通信内容が暗号化されるので、途中で盗み見ても意味不明
2. **改ざん検知**: 通信内容が途中で書き換えられたら検出できる
3. **なりすまし防止**: サーバーが本物かを **証明書** で確認する（`example.com` を名乗っているのは本当に example.com の所有者か）

**実例**: 公衆 Wi-Fi（カフェや空港）に繋いだとき、もし HTTP（暗号化なし）で銀行のサイトに入ったら、同じ Wi-Fi にいる人が通信を覗ける可能性があります。HTTPS（暗号化あり）なら、覗かれても中身は読めません。今では大半のサイトが HTTPS になっています。

#### TLS ハンドシェイク

TCP の 3-way ハンドシェイクが終わった **あと** に、さらに TLS のハンドシェイクが走ります。

1. クライアントが「対応している暗号方式」を伝える
2. サーバーが「これでいこう」と決めて、**証明書** を渡す
3. クライアントが証明書を検証する（信頼できる認証局のサインがあるか）
4. お互いだけが知っている共通鍵を作る
5. 以降の通信は共通鍵で暗号化

**実例**: ブラウザの URL バー左の **鍵マーク** をクリックすると、そのサイトが使っている証明書を見られます。誰が発行したか（Let's Encrypt とか DigiCert とか）、有効期限はいつまでか、などが書かれています。

#### 証明書エラー

「この接続は安全ではありません」とブラウザが警告するのは、

- 証明書の有効期限が切れている
- 証明書のドメインと、アクセスしているドメインが一致しない
- 信頼できない認証局が発行した（自己署名証明書など）

のどれかです。社内システムや個人開発のサーバーで自己署名証明書を使うとよく見る画面です。

### 10. CORS の予告

Chapter 5 で詳しくやりますが、HTTP の話の流れで一度触れておきます。

**CORS（Cross-Origin Resource Sharing）** は「あるドメインの JS が、別のドメインの API を叩いていいか」を制御する仕組みです。ブラウザのセキュリティ機能で、サーバー側が `Access-Control-Allow-Origin` などのヘッダーで明示的に許可を出さないと、JS からのリクエストはブラウザにブロックされます。

「自分で作った API をブラウザの JS から叩いたら、なぜか CORS エラーが出る」というのは Web 開発の通過儀礼です。Chapter 5 でわざと出してから直します。

> このセクションは「最初に手を動かしたとき**ほぼ確実に踏む**罠」のリスト。コードが動かないとき、まずここを疑う。

- **ヘッダーとボディの間の空行を忘れる**: 生ソケットで HTTP を書くと一度はやらかすミス。<mark>`\r\n\r\n` がヘッダー終端の合図</mark>
- **改行は `\r\n`**: HTTP の仕様では LF 単体（`\n`）ではなく <mark>CRLF（`\r\n`）</mark>です。`\n` だけだと動かない実装もある
- **GET にボディを付けない**: 仕様上は付けられるが、無視されたり拒否されたりするので避ける。パラメータは URL の query で渡す
- <mark>**`200 OK` は「HTTP として成功」、業務として成功とは限らない**</mark>: API のエラーがボディに JSON で入ってくることがある
- **HTTPS は HTTP の別物ではない**: 上で書いたとおり、暗号化の層を 1 枚かぶせただけ。HTTP の知識は HTTPS でもそのまま使える
- <mark>**Cookie は同じドメインにしか飛ばない**</mark>: `a.com` のサイトで保存した Cookie は `b.com` には送られない。これを覚えておくと CORS や CSRF の話が分かりやすくなる

## 重要な単語まとめ

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **HTTP リクエスト / レスポンス** | クライアント要求 / サーバー応答のテキスト | 開発者ツールの Network タブの 1 行 1 行 |
| **リクエストライン** | リクエストの 1 行目（メソッド + パス + バージョン） | `GET /search?q=web HTTP/1.1` |
| **ステータスライン** | レスポンスの 1 行目（バージョン + コード + 理由） | `HTTP/1.1 200 OK` |
| **ヘッダー** | 本文に添えるメタ情報の `名前: 値` の行 | `Content-Type`, `User-Agent`, `Cookie` |
| **ボディ** | 本文。HTML / JSON / 画像など何でも入る | レスポンスの中身そのもの |
| **GET / POST / PUT / PATCH / DELETE** | サーバーに何を頼むか（読む/作る/上書き/部分更新/消す） | ページ表示 / 投稿 / プロフ編集 / 削除 |
| **冪等（idempotent）** | 何度実行しても結果が同じ | DELETE を 2 回押しても「消えてる」は変わらない |
| **2xx / 3xx / 4xx / 5xx** | 成功 / リダイレクト / クライアントエラー / サーバーエラー | 200 OK, 404, 500 |
| **404 / 403 / 401** | ない / 禁止 / 名乗ってない | リンク切れの定番画面 |
| **500 / 502 / 503** | アプリのバグ / 前段が繋げない / 意図的に拒否 | サーバー障害の原因切り分けに使う |
| **Content-Type** | ボディの形式 | `application/json`, `text/html` |
| **Cookie / Set-Cookie** | 状態を保持するための小さなテキスト | ログインの継続、カートの中身 |
| **HttpOnly / Secure / SameSite** | Cookie のセキュリティ属性 | セッション ID は HttpOnly + Secure が定番 |
| **セッション** | サーバー側で同じユーザーの連続アクセスをまとめる概念 | ログイン状態の維持 |
| **ステートレス** | サーバーは前回の通信を覚えない | だから Cookie が必要 |
| **HTTPS / TLS** | HTTP に暗号化レイヤをかぶせたもの | URL バーの鍵マーク |
| **証明書** | サーバーが本物であることを示す書類 | 鍵マークをクリックすると見られる |
| **HTTP/1.1 / HTTP/2 / HTTP/3** | プロトコルのバージョン | 開発者ツールの Protocol 列 |
| **CORS** | 別ドメインへの JS からの通信を制御する仕組み | Ch5 でわざと出して直す |
