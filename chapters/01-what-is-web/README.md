# Chapter 1: Web とは何か

> Status: 完了

- 「Web」と「インターネット」は別物だということ

## 本文

### 1. Web とインターネットは違う

実は混同されがちですが、<mark>**インターネット**と **Web** は別の概念</mark>です。

- **インターネット** は、世界中のコンピュータが共通の約束（TCP/IP）でつながりあう、世界規模の **ネットワークの集合体**。電線・光ファイバー・無線などの物理的な配線はその一部にすぎず、「道路網」のイメージが近い。
- **Web（World Wide Web）** は、そのインターネットという道路網の上を走るサービスの **ひとつ**。電子メール・ファイル転送・オンラインゲームなどと同じく、インターネット上で動くサービスの一種類にすぎません（メールには SMTP、ファイル転送には FTP、Web には HTTP という「やり取りの作法」が裏で使われている、という関係）。

ただし規模と影響力が桁違いなので、多くの人にとっては Web ＝インターネットのように感じられます。包含関係を図にすると、

```
インターネット ⊃ Web ⊃ あなたが今見ている Web サイト
```

「ネットが遅い」と言うときの “ネット” は、厳密にはインターネットの話なのか Web の話なのか、エンジニアと話すときは少しだけ意識すると会話がずれません。

### 2. Web の短い歴史

Web は 1989 年、イギリス人物理学者ティム・バーナーズ＝リーが、欧州原子核研究機構（CERN）で働いていたときに提案したのが始まりです。研究者たちが論文や実験データを互いに参照しやすくするための仕組みとして発案されました。

転機になったキーワードは **ハイパーテキスト（hypertext）**。文書から別の文書へリンクで飛べる仕組みで、これが「Web（クモの巣）」という名前の由来です。

ざっくり流れを押さえておくと、

- **1991 年**: 世界初の Web サイトが公開される（今も `info.cern.ch` にアーカイブが残っています）
- **1990 年代後半**: ブラウザ戦争（Netscape vs Internet Explorer）。HTML が急速に普及
- **2000 年代**: Web 2.0。ユーザーが「見る側」から「投稿する側」に回る（ブログ・SNS・Wikipedia）
- **2010 年代以降**: スマートフォン・クラウド・API エコノミー。Web アプリが OS 上のアプリを置き換え始める
- **現在**: AI が Web を読み書きする時代に入りつつある

非エンジニアの感覚としては「Web は最初から双方向で当たり前」ですが、当初は **読むだけの一方通行メディア** だった、というのが重要なポイントです。

### 3. クライアントとサーバー

Web を語るうえで避けて通れないのが **クライアント・サーバーモデル** です。

- **クライアント**: 情報を「お願いする側」。あなたのスマホや PC で動いているブラウザがこれにあたる
- **サーバー**: 情報を「提供する側」。世界のどこかでずっと起動しているコンピュータ

あなたがブラウザで URL を開くたび、ブラウザ（クライアント）が「このページください」とサーバーにお願いし、サーバーがそれに応じて返事をしています。このやり取りの往復のことを、

- <mark>**リクエスト（request）** = 要求</mark>
- <mark>**レスポンス（response）** = 応答</mark>

と呼びます。本ハンドブックでは何度も出てくる重要な対義語ペアです。

### 4. Web を支える 3 つの基本技術

> [!CAUTION]
> Web は、巨大で複雑に見えて、技術の根っこをたどると <mark>**URL・HTTP・HTML の 3 本**</mark> に行き着く。この 3 つは Chapter 2〜4 の主役になるので、名前だけでもここで頭に入れておく。

| 技術 | 役割 | 一言で言うと |
|---|---|---|
| **URL** | リソース（情報）の住所 | 「どこにあるか」 |
| **HTTP** | やり取りの作法 | 「どう話すか」 |
| **HTML** | 文書の構造 | 「中身をどう書くか」 |

これらが揃って初めて、「住所を指定して」「決まった手順で問い合わせて」「返ってきた中身を読む」という一連の動作が成立します。

このハンドブックでは、Chapter 2 でネットワーク全体、Chapter 3 で HTTP、Chapter 4 でデータ形式（HTML や JSON）と、それぞれを一段ずつ深掘りしていきます。

### 5. ブラウザでページが表示されるまで

URL バーに `https://example.com` と入力して Enter を押した瞬間、裏では大まかに次のことが起きています。

1. **名前解決**: `example.com` という名前を、コンピュータが分かる IP アドレス（例: `93.184.216.34`）に変換する。これが DNS（Domain Name System）の仕事
2. **接続**: ブラウザがそのIPアドレスのサーバーに通信路を確保する（TCP 接続、最近はその上で TLS による暗号化）
3. **リクエスト送信**: 「このページをください」という HTTP リクエストを送る
4. **レスポンス受信**: サーバーが HTML や画像・CSS・JavaScript を返してくる
5. **描画（レンダリング）**: ブラウザが受け取った HTML を解釈して画面に絵を描く

この 5 段階は Web の「お決まりの流れ」です。Chapter 2 では 1〜2 を、Chapter 3 では 3〜4 を、実際に Python から手で叩いて体感していきます。

### 6. Web サイト・Web アプリ・Web サービス

似た言葉を整理しておきます。

- **Web サイト**: 主に「見る」ためのページ群（ブログ、コーポレートサイトなど）
- **Web アプリ**: ブラウザの中で動く「アプリ」（Gmail、Google Maps、X など）
- **Web サービス**: 機能を **API**（プログラム同士が呼び出しあうための窓口）として他のシステムに提供するもの（天気 API、決済 API など）

境界は曖昧で、最近のサービスはほぼ全部「Web アプリ＋API」のセットです。Chapter 5 で、自分でも小さな Web アプリと API を作ります。

### 7. 静的サイトと動的サイト

- **静的サイト**: あらかじめ HTML が用意されていて、誰が見ても同じものが返る
- **動的サイト**: アクセスした人や時刻に応じて、サーバーがその場で HTML を組み立てて返す

X（旧 Twitter）のタイムラインが人によって違うのは、サーバーがログインユーザーごとに動的に内容を作っているからです。一方、企業のコーポレートサイトのトップページは誰が見ても同じ静的サイトであることが多い。

「動的に見える」けれど中身は「静的＋ JavaScript の後付け」というハイブリッド構成もよくあり、現代の Web ではこの境界もぼやけています。

## 手を動かすパート

この章では本格的に手を動かしません。文章を読んで全体像をつかむことが目的です。

代わりに、いまブラウザで適当な Web サイトを開いて、`F12` キー（または右クリック →「検証」）で **開発者ツール** を開いてみてください。

1. 上部のタブから **Network** を選ぶ
2. ページを再読み込み（`Ctrl + R` / `Cmd + R`）
3. 一覧にずらっと並ぶ通信ログを眺める

ブラウザが裏で何十・何百ものリクエストを飛ばしていることが見えるはずです。「あ、これが後の章で深掘りするやつか」という感触を持っておけば十分です。

## つまずきポイント

> [!CAUTION]
> 入門時点で **誤解しがちな概念のリスト**。とくに 2 番目（HTML はプログラミング言語ではない）は次章以降の話のベースになるので必ず押さえる。

- **「Web」と「インターネット」を厳密に区別する場面はそんなに多くない**: 日常会話では混ぜて使っても通じます。ただしエンジニアと話す文脈、とくに「ネットワーク」「プロトコル」という言葉が出てきたときは、自分が今どっちの層の話をしているのか意識すると混乱しません。
- <mark>**HTML は「プログラミング言語」ではない**</mark>: 入門書でよく注意されるポイント。HTML は構造を示すマークアップ言語で、`if` や `for` といった処理は書けません。動的な処理は JavaScript やサーバー側の言語（Python など）が担当します。
- **クライアント・サーバーモデルは「2 点間」だけが本質ではない**: 実際には CDN・キャッシュ・ロードバランサ・プロキシなど、間にたくさんの中継地点が挟まっています。最初は 2 点間と捉えて OK、慣れたら「実はもっと多段構造」と上書きしてください。
- **「Web 業界」と「Web 技術」も別の言葉**: Web 業界＝ Web を使ったビジネスをしている会社・人のこと。本ハンドブックで扱うのは **Web 技術**、つまり仕組みのほうです。

## 重要な単語まとめ

次の章以降で繰り返し出てくる「これを知らないと話が通じない」言葉だけ抜き出しておきます。今は「なんとなくの輪郭」がつかめていれば十分。詳しい定義は各章で深掘りします。

| 用語 | 一言で言うと | 身近な実例 |
|---|---|---|
| **インターネット** | 世界中のコンピュータが共通の約束でつながりあうネットワークの集合体。「道路網」のイメージ | Wi-Fi、4G/5G、光ファイバーの先にある「外の世界」全部 |
| **Web（WWW）** | インターネット上で動くサービスのひとつ。文書同士をリンクでつなぐ仕組み | ブラウザで開く YouTube・Wikipedia・X など |
| **ハイパーテキスト** | 文書から別の文書へリンクで飛べる仕組み。Web の「Web（クモの巣）」の由来 | Wikipedia の青い見出しリンクをたどっていく感じ |
| **クライアント** | 情報を「お願いする側」 | あなたのスマホ・PC で動いているブラウザ |
| **サーバー** | 情報を「提供する側」。常時起動しているコンピュータ | YouTube の動画を配っているコンピュータ |
| **リクエスト / レスポンス** | クライアントの「要求」とサーバーの「応答」のペア | LINE の「メッセージ送信 → 既読」のような往復 |
| **URL** | リソースの「住所」。`https://example.com/path` の形 | ブラウザのアドレスバーに入れるあれ |
| **HTTP** | Web 上でのやり取りの「作法（プロトコル）」。Chapter 3 の主役 | URL の先頭の `http://` / `https://` |
| **HTML** | Web ページの「構造」を書くためのマークアップ言語。プログラミング言語ではない | `Ctrl + U` で見えるあのタグの嵐 |
| **DNS** | ドメイン名（`example.com`）を IP アドレスに変換する仕組み。「名前解決」 | `youtube.com` と入れるだけで動画が観られる裏の仕組み |
| **IP アドレス** | ネットワーク上のコンピュータの「番地」。数字の羅列 | スマホの Wi-Fi 詳細に出てくる `192.168.x.x` |
| **TCP** | 確実にデータを届けるための通信方式。HTTP はこの上で動いている | LINE のテキスト送信のような「ちゃんと届く」系 |
| **TLS / HTTPS** | 通信を暗号化する仕組み。HTTPS は「TLS で守られた HTTP」 | URL バーの鍵マーク |
| **レンダリング** | ブラウザが受け取った HTML を画面上の絵に変換する処理。「描画」 | ページを開いた瞬間に文字や画像が並ぶあれ |
| **静的サイト / 動的サイト** | 誰が見ても同じものが返るのが静的。アクセスごとに中身を組み立てるのが動的 | 企業のコーポレートサイト / X のタイムライン |
| **Web サイト / Web アプリ / Web サービス** | 見るためのページ群 / ブラウザ内で動くアプリ / API として機能を提供するもの | ブログ / Gmail / 天気 API |
| **API** | プログラムからプログラムへ機能を呼び出すための窓口。Chapter 5 で自作する | 天気アプリの裏で叩いている気象データの API |
