## きっかけ

読んでいて出てきた「なんで？」を集めました。

- TCP接続とTLSによる暗号化の意味は？
- なぜ最初からコンピュータが名前を分かるようにしなかったのか？機械語の0と1と関係ある？
- レンダリング（描画）は「レスポンスをユーザーのPCで解釈すること」で良い？
- プロトコルとは何か？

---

## 1. プロトコル — 通信の「共通ルール」

**プロトコル**とは「通信する者どうしが守る、あらかじめ決めた約束事・手順」のことです。

### 身近な例

LINEでメッセージを送るとき、送る側・受け取る側が同じアプリの同じルールで動いているから成立します。送る側が「テキスト」、受け取る側が「音声のみ解釈」だったら通信は破綻します。プロトコルはこの「共通言語」です。

### Web で登場する主なプロトコル

| プロトコル | 担当する約束事 |
|---|---|
| HTTP | ブラウザとサーバーが「リクエスト/レスポンス」をやり取りする手順 |
| TCP | データを順番通り・確実に届ける手順 |
| TLS | 暗号化の方式を合意して通信する手順 |
| DNS | 名前をIPアドレスに変換する手順 |

### なぜ必要か

世界中の異なるメーカーのPC・スマホ・サーバーが通信できるのは、共通プロトコルがあるからです。英語が国際共通語として機能するのと同じ理屈です。

## 2. IPアドレスと名前 — 機械語の0と1がルーツ

### コンピュータは「数値の機械」

機械語が0と1であるとおり、コンピュータはすべてを数値として処理します。IPv4のIPアドレスは実は **32ビットの2進数** です。

```
93.184.216.34
= 01011101.10111000.11011000.00100010（2進数）
```

人間が `93.184.216.34` と書いているのは「32ビットの数値を読みやすく10進数で表示したもの」に過ぎません。

### なぜ名前ではなく数値なのか

ルーターはパケットを受け取るたびに「どの方向に転送するか」を大量に高速計算します。固定長の数値ならその計算が一瞬ですが、`example.com` のような可変長の文字列だとまず文字列として解析する処理が必要で効率が落ちます。

### 名前の概念は最初からあった

インターネット黎明期には `hosts.txt` という1枚のファイルに「この名前 → このIP」の全対応表を書き、世界中のコンピュータが同じファイルを共有していました。コンピュータが爆発的に増えて管理が限界になったため、**1983年にDNS（Domain Name System）が作られました**。

| | 機械（ルーター） | 人間 |
|---|---|---|
| 得意なもの | 数値（固定長ビット列）の計算 | 意味のある名前 |
| 使うもの | `93.184.216.34` | `example.com` |

**DNS は「人間が使いやすい名前 ↔ コンピュータが処理しやすい数値」の翻訳係** です。

## 3. TCP接続とTLS — 「通信路の確立」と「金庫」

TCP と TLS は別々の役割を持ち、順番に動きます。

### TCP接続（通信路の確立）

3-wayハンドシェイクで相手と「話しかけるよ」→「聞こえてるよ」→「了解」を確認し合い、コネクションを確立します。ただし **この時点ではデータは平文（そのまま）** で流れています。

### TLSによる暗号化（金庫）

TCP接続の上に「暗号化の層」を追加します。

```
1. TCP 3-wayハンドシェイク（コネクション確立）
2. TLSハンドシェイク（暗号方式の合意・証明書確認・共通鍵の共有）
3. 暗号化された状態でHTTPのやり取り → これが HTTPS
```

### 例え

| | 例え |
|---|---|
| TCP接続 | 電話がつながった状態（相手の声は聞こえるが、傍受もされうる）|
| TLS暗号化 | 「どちらも持っている暗号ブック」で喋る（盗聴されても内容が意味不明になる）|

**TCP = 通信路の確立、TLS = その上の金庫。HTTPSは金庫の中でHTTPをやり取りする状態です。**

---

## 4. レンダリング（描画） — テキストを画面に変える

サーバーからのレスポンスはただの**テキスト**（HTMLやCSS）です。「レスポンスをユーザーのPCで解釈して画面に出力する」がレンダリングの正確な意味です。

```
レスポンス（テキスト）
  └─ ブラウザが解析（パース）
       ├─ HTML → どんな要素があるか
       ├─ CSS  → どんな見た目か
       └─ 画面に描く（レンダリング）
```

### ポイント

- サーバーは**テキストを送るだけ**。「画面を作る」のはブラウザ（PC側）
- 同じHTMLでも、ブラウザの種類やバージョンで見た目が微妙に変わることがあるのはこのため
- JavaScriptを使うと、レスポンス受信後にPC側でさらにHTMLを書き換えて画面を更新できる（SPAの仕組み）

## 重要な単語まとめ

| 単語 | 一言定義 | 身近な実例 |
|---|---|---|
| プロトコル | 通信の共通ルール・約束事 | LINEの送受信ルール、電話の「もしもし」 |
| DNS | 名前→IPアドレスの翻訳システム | 電話帳（名前で引いて番号を調べる） |
| TCP | データを確実・順番通りに届ける仕組み | 宅配便の追跡・再配達システム |
| TLS | TCP上の暗号化レイヤー | 封筒に鍵をかける |
| HTTPS | TLSで暗号化されたHTTP通信 | URL欄の錠前マーク |
| レンダリング | テキストを画面上のピクセルに変換する処理 | 設計図（HTML）を見て家（画面）を建てる |
